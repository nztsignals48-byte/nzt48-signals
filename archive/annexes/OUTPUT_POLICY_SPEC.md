## 1. Objective

This document defines **binding rules for ALL system outputs** produced by the NZT-48 Trading Engine. Every outbound message, report, and data stream must pass through the gates defined herein before reaching any external consumer.

Covered output channels:

| Channel | Transport | Consumer |
|---|---|---|
| Telegram Signals | Telegram Bot API | Operator mobile / desktop |
| PDF Reports | Email attachment / local filesystem | Operator review |
| War Room | WebSocket (dashboard :3001) | Operator browser |
| Premarket Briefs | Telegram + PDF | Operator pre-session |

**Design Principle:** The system is **fail-closed**. When in doubt, suppress. A missed signal costs opportunity; a bad signal costs capital. The asymmetry is absolute.

**Authoritative Scope:** These rules apply to every code path that calls `telegram_send()`, `pdf_generate()`, `warroom_push()`, or any wrapper, alias, or future equivalent. No output path may bypass these gates.

---

## 2. Scope

This policy governs:

- All Telegram messages sent by the NZT-48 engine (signals, alerts, status updates, error notifications)
- All PDF reports generated by the delivery subsystem (`pdf_v2_momentum.py`, `pdf_v2_risk.py`, and any future report modules)
- All War Room WebSocket pushes to the Next.js dashboard
- All premarket briefs regardless of delivery mechanism
- Any future output channel added to the system

This policy does NOT govern:

- Internal logging (stdout, file logs, Docker logs) -- these are unrestricted
- Health check endpoints (`/health`, `/status`) -- these always respond
- Metrics/monitoring endpoints -- these always respond

## 3. Definitions

| Term | Definition |
|---|---|
| **Signal** | Any actionable trading message containing a ticker, direction, and/or score |
| **Gate** | A validation checkpoint that must pass before output is released |
| **Suppress** | Silently block output from reaching the consumer; log the suppression event |
| **Block** | Actively prevent output and generate an operator-visible alert |
| **TTL** | Time-to-live; the maximum age of a data field before it is considered stale |
| **Composite Score** | The final weighted score assigned to a signal by the scoring engine |
| **Regime** | The current market regime classification (e.g., TRENDING_UP, MEAN_REVERTING, VOLATILE, LOW_VOL) |
| **Kill Switch** | A master toggle that suppresses ALL outbound signals when activated |
| **Quiet Mode** | A time-limited state after restart where Telegram signals are held |
| **DEGRADED DATA** | A watermark/banner indicating that data quality checks did not fully pass |
| **UNVERIFIED** | A flag indicating a data point has exceeded plausibility thresholds |
| **Operator** | The human administrator of the NZT-48 system |

## 4. Rule 1: Fail-Closed Default

### Statement

If any data quality check returns anything other than `ALL_CHECKS_PASSED`, suppress all outbound Telegram signals. PDF and War Room may continue to display but MUST include a `DEGRADED DATA` watermark.

### Specification

```
FUNCTION pre_send_quality_gate():
    result = run_all_data_quality_checks()

    IF result.status != "ALL_CHECKS_PASSED":
        telegram.suppress_all()
        pdf.set_watermark("DEGRADED DATA")
        warroom.set_banner("DEGRADED DATA — Quality checks failed: {result.failures}")
        log.warning("OUTPUT_POLICY: Fail-closed triggered", checks=result.details)
        RETURN GATE_BLOCKED

    RETURN GATE_PASSED
```

### Data Quality Checks (minimum set)

| Check ID | Description | Failure Condition |
|---|---|---|
| `DQ-001` | Price feed connectivity | Any primary feed returns HTTP != 200 or timeout > 10s |
| `DQ-002` | Price feed freshness | Last price update older than 120s during market hours |
| `DQ-003` | Regime engine output | Regime classifier returned null, error, or has not run in current session |
| `DQ-004` | Scoring engine output | Scoring pipeline returned null or threw exception on last run |
| `DQ-005` | ISA ticker coverage | Fewer than 80% of ISA fund tickers returning valid data |
| `DQ-006` | System clock sync | System clock drift exceeds 5s from NTP reference |

### Behaviour Matrix

| Quality Status | Telegram | PDF | War Room | Premarket Brief |
|---|---|---|---|---|
| `ALL_CHECKS_PASSED` | SEND | GENERATE | PUSH | SEND |
| Any check failed | SUPPRESS | GENERATE + WATERMARK | PUSH + BANNER | SUPPRESS |
| Multiple checks failed | SUPPRESS | GENERATE + WATERMARK | PUSH + BANNER | SUPPRESS |
| Quality engine itself errored | SUPPRESS | DEFER | PUSH + "QUALITY ENGINE DOWN" | SUPPRESS |

### Implementation Notes

- The quality gate runs BEFORE every Telegram send, not on a timer. This ensures no stale pass/fail result is used.
- PDF generation may proceed with the watermark because the operator may still want to review the data visually. The watermark must be unambiguous and visible on every page.
- War Room banner must persist until the next quality check passes.

## 5. Rule 2: Staleness Gate

### Statement

Every outbound message must include an `as-of` timestamp for its primary data source. If any data field is older than its TTL (defined per field), the message is BLOCKED.

### Field TTL Definitions

| Data Field | TTL (Market Hours) | TTL (Off-Hours) | Notes |
|---|---|---|---|
| Last trade price | 120 seconds | 24 hours | LSE closes 16:30 UK; off-hours = after close |
| Bid/Ask spread | 120 seconds | 24 hours | |
| Volume (intraday) | 300 seconds | 24 hours | |
| ATR (daily) | 24 hours | 48 hours | Recalculated at session open |
| Regime classification | 600 seconds | 24 hours | Must reflect current session |
| Composite score | 300 seconds | N/A | Scores are not sent off-hours |
| Sector rotation data | 1 hour | 48 hours | |
| Correlation matrix | 1 hour | 48 hours | |
| Volatility regime | 600 seconds | 24 hours | |
| Futures/overnight data | 600 seconds | 2 hours | Tighter off-hours TTL for futures |

### Specification

```
FUNCTION staleness_gate(message):
    FOR EACH data_field IN message.referenced_fields:
        field_age = NOW() - data_field.timestamp
        ttl = get_ttl(data_field.name, is_market_hours())

        IF field_age > ttl:
            log.warning("OUTPUT_POLICY: Staleness gate BLOCKED",
                field=data_field.name,
                age_seconds=field_age.total_seconds(),
                ttl_seconds=ttl.total_seconds())
            RETURN GATE_BLOCKED, {
                "reason": "STALE_DATA",
                "field": data_field.name,
                "age": field_age,
                "ttl": ttl
            }

    message.metadata["as_of"] = min(f.timestamp for f in message.referenced_fields)
    RETURN GATE_PASSED
```

### Message Format

Every outbound Telegram signal must include:

```
[SIGNAL] BUY QQQ3.L | Score: 72 | Target: +2.00%
as-of: 2026-02-27T09:31:15Z | regime: TRENDING_UP
```

Every PDF must include in its header:

```
Data as-of: 2026-02-27T09:30:00Z
Freshness: ALL FIELDS WITHIN TTL
```

War Room WebSocket frames must include an `as_of` field in the JSON payload.

## 6. Rule 3: Magnitude Gate

### Statement

Any single-ticker return exceeding +/-30% in a single session, or any futures/overnight move exceeding +/-8%, must be flagged as `UNVERIFIED` and held for manual review before sending.

### Specification

```
FUNCTION magnitude_gate(signal):
    IF signal.type == "EQUITY" OR signal.type == "ETP":
        IF abs(signal.session_return_pct) > 30.0:
            signal.flag = "UNVERIFIED"
            signal.hold_for_review = TRUE
            log.critical("OUTPUT_POLICY: Magnitude gate HOLD",
                ticker=signal.ticker,
                return_pct=signal.session_return_pct,
                threshold=30.0)
            notify_operator("MAGNITUDE_HOLD: {ticker} showing {return}% — manual review required")
            RETURN GATE_HELD

    IF signal.type == "FUTURES" OR signal.context == "OVERNIGHT":
        IF abs(signal.move_pct) > 8.0:
            signal.flag = "UNVERIFIED"
            signal.hold_for_review = TRUE
            log.critical("OUTPUT_POLICY: Magnitude gate HOLD (futures/overnight)",
                ticker=signal.ticker,
                move_pct=signal.move_pct,
                threshold=8.0)
            notify_operator("MAGNITUDE_HOLD: {ticker} overnight move {move}% — manual review required")
            RETURN GATE_HELD

    RETURN GATE_PASSED
```

### Thresholds

| Asset Class | Threshold | Action |
|---|---|---|
| Equity / ETP (single session) | +/-30% | HOLD + UNVERIFIED flag |
| Futures / Overnight gap | +/-8% | HOLD + UNVERIFIED flag |
| Leveraged ETP (3x) | +/-30% on underlying equivalent (i.e., +/-90% on 3x product) | HOLD + UNVERIFIED flag |

### Rationale

Moves of this magnitude are statistically implausible under normal conditions. They typically indicate:

- Data feed error (stale close vs. current price)
- Corporate action (stock split, reverse split, delisting)
- Genuine black swan event

In all three cases, automatic signal delivery is dangerous. The cost of a 5-minute delay for manual verification is negligible compared to the cost of acting on corrupt data.

### Manual Review Process

1. System sends operator a Telegram notification with the held signal details.
2. Operator reviews via War Room or direct data source.
3. Operator issues either:
   - `POST /api/override/magnitude-release` with signal ID and reason, OR
   - `POST /api/override/magnitude-discard` with signal ID and reason.
4. Released signals are sent with an `[OPERATOR VERIFIED]` tag.
5. Discarded signals are logged and permanently suppressed.

## 7. Rule 4: Consistency Gate

### Statement

Before any Telegram send, cross-check that the signal's regime label matches the current system regime. If they differ, BLOCK the signal and log a `REGIME_MISMATCH` event.

### Specification

```
FUNCTION consistency_gate(signal):
    current_regime = regime_engine.get_current()
    signal_regime = signal.regime_at_generation

    IF current_regime IS NULL:
        log.error("OUTPUT_POLICY: Regime engine returned NULL — blocking")
        RETURN GATE_BLOCKED, {"reason": "REGIME_NULL"}

    IF signal_regime != current_regime:
        log.warning("OUTPUT_POLICY: Regime mismatch",
            signal_regime=signal_regime,
            current_regime=current_regime,
            signal_id=signal.id,
            ticker=signal.ticker)
        event_log.record("REGIME_MISMATCH", {
            "signal_id": signal.id,
            "signal_regime": signal_regime,
            "current_regime": current_regime,
            "timestamp": NOW(),
            "ticker": signal.ticker
        })
        RETURN GATE_BLOCKED, {"reason": "REGIME_MISMATCH"}

    RETURN GATE_PASSED
```

### Why This Matters

Signals are generated asynchronously. Between the moment a signal is scored and the moment it reaches the Telegram send queue, the regime may have shifted. A signal generated under `TRENDING_UP` that arrives during `VOLATILE` may be stale or outright dangerous.

### Regime Transition Grace Period

To avoid excessive blocking during legitimate regime transitions:

- If the regime changed within the last 60 seconds, the signal is BLOCKED but re-queued for one retry after 60 seconds.
- If on retry the regimes still mismatch, the signal is permanently BLOCKED.
- No more than one retry per signal.

### Mismatch Event Schema

```json
{
    "event_type": "REGIME_MISMATCH",
    "signal_id": "sig_20260227_093115_QQQ3L",
    "signal_regime": "TRENDING_UP",
    "current_regime": "VOLATILE",
    "ticker": "QQQ3.L",
    "timestamp": "2026-02-27T09:31:45Z",
    "action": "BLOCKED",
    "retry_scheduled": true
}
```

## 8. Rule 5: Score Floor

### Statement

No signal with composite score < 10 may be sent via any channel. Signals with score 0 are always suppressed.

### Specification

```
FUNCTION score_floor_gate(signal):
    IF signal.composite_score == 0:
        log.debug("OUTPUT_POLICY: Score 0 — always suppressed", ticker=signal.ticker)
        RETURN GATE_SUPPRESSED

    IF signal.composite_score < 10:
        log.info("OUTPUT_POLICY: Score below floor",
            ticker=signal.ticker,
            score=signal.composite_score,
            floor=10)
        RETURN GATE_SUPPRESSED

    RETURN GATE_PASSED
```

### Channel-Specific Behaviour

| Score Range | Telegram | PDF | War Room | Premarket Brief |
|---|---|---|---|---|
| 0 | SUPPRESS | SUPPRESS | SUPPRESS | SUPPRESS |
| 1 -- 9 | SUPPRESS | SUPPRESS | SUPPRESS | SUPPRESS |
| 10 -- 29 | SEND (low confidence tag) | INCLUDE | INCLUDE | INCLUDE |
| 30 -- 69 | SEND | INCLUDE | INCLUDE | INCLUDE |
| 70 -- 100 | SEND (high conviction tag) | INCLUDE (highlighted) | INCLUDE (highlighted) | INCLUDE (highlighted) |

### Score 0 Special Case

A score of exactly 0 indicates one of:

- Ticker is excluded from the active universe
- Scoring engine explicitly zeroed the ticker (e.g., halted, delisted)
- Data insufficient to produce a score

In all cases, score 0 signals are silently suppressed. No log noise beyond `DEBUG` level.

## 9. Rule 6: Rate Governance

### Statement

All output channels are rate-limited to prevent spam, abuse, and resource exhaustion.

### Telegram Rate Limits

| Limit | Value | Action on Breach |
|---|---|---|
| Per-minute cap | 5 messages/min | Queue excess; send when slot available |
| Per-hour cap | 30 messages/hour | Queue excess; send when slot available |
| Spam kill trigger | 10 messages in any 60-second window | AUTO-PAUSE all Telegram sends for 15 minutes |
| Daily soft cap | 200 messages/day | Log warning; no block |

### Spam Kill Specification

```
FUNCTION telegram_rate_governor(message):
    recent_60s = count_messages_sent(last_60_seconds)
    recent_hour = count_messages_sent(last_3600_seconds)

    IF recent_60s >= 10:
        log.critical("OUTPUT_POLICY: Spam kill triggered — 15min pause",
            count_60s=recent_60s)
        telegram.pause(duration=900)  # 15 minutes
        notify_operator("SPAM KILL: Telegram paused 15min. {recent_60s} msgs in 60s.")
        RETURN RATE_BLOCKED

    IF recent_60s >= 5:
        queue.enqueue(message, priority=message.score)
        log.info("OUTPUT_POLICY: Per-minute rate limit — queued")
        RETURN RATE_QUEUED

    IF recent_hour >= 30:
        queue.enqueue(message, priority=message.score)
        log.info("OUTPUT_POLICY: Per-hour rate limit — queued")
        RETURN RATE_QUEUED

    RETURN RATE_PASSED
```

### PDF Rate Limits

| Limit | Value | Action on Breach |
|---|---|---|
| Per scheduled slot | 1 PDF per slot | Discard duplicate generation requests |
| Minimum interval | 30 minutes between PDFs | Queue if too soon |

Scheduled PDF slots are defined in `config/settings.yaml`. Typical schedule:

- Premarket PDF: 06:30 UK
- Momentum PDF (PDF1): 09:15 UK (after LSE open)
- Risk PDF (PDF2): 16:45 UK (after LSE close)

### War Room Rate Limits

| Limit | Value | Action on Breach |
|---|---|---|
| WebSocket frame rate | Max 2 updates/sec | Throttle; coalesce updates |
| Burst allowance | 10 frames in 2 seconds | Allowed (for regime change events) |
| Sustained burst | >2/sec for >10 seconds | Throttle to 1/sec; log warning |

### Implementation Notes

- Rate counters use sliding windows, not fixed windows. This prevents edge-of-window bursts.
- The spam kill auto-pause is a safety mechanism against runaway loops. It is NOT a substitute for fixing the root cause.
- After a spam kill, the operator must investigate the cause. Repeated spam kills (3 in 24 hours) trigger a system-wide kill switch activation.

## 10. Rule 7: Restart Hygiene

### Statement

On service restart, enter a 5-minute `QUIET MODE` where no Telegram signals are sent. War Room displays a `SYSTEM RESTARTING` banner. PDFs scheduled during quiet mode are deferred.

### Specification

```
FUNCTION on_service_start():
    state.quiet_mode = TRUE
    state.quiet_mode_start = NOW()
    state.quiet_mode_duration = 300  # 5 minutes

    warroom.set_banner("SYSTEM RESTARTING — Quiet mode until {quiet_end}")
    log.info("OUTPUT_POLICY: Entering quiet mode for 300s")

    # Restore kill switch state (see Rule 8)
    restore_persisted_state()

    # Schedule quiet mode exit
    scheduler.schedule(exit_quiet_mode, delay=300)

FUNCTION exit_quiet_mode():
    state.quiet_mode = FALSE
    warroom.clear_banner("SYSTEM RESTARTING")
    log.info("OUTPUT_POLICY: Quiet mode ended — outputs enabled")

    # Process any queued signals that passed all gates
    process_deferred_queue()

FUNCTION quiet_mode_gate(message):
    IF state.quiet_mode:
        IF message.channel == "TELEGRAM":
            queue.defer(message)
            log.info("OUTPUT_POLICY: Quiet mode — Telegram deferred", signal=message.id)
            RETURN GATE_DEFERRED

        IF message.channel == "PDF":
            queue.defer(message)
            log.info("OUTPUT_POLICY: Quiet mode — PDF deferred", signal=message.id)
            RETURN GATE_DEFERRED

        IF message.channel == "WARROOM":
            # War Room still receives data, but with banner
            RETURN GATE_PASSED

    RETURN GATE_PASSED
```

### Rationale

On restart, the system must:

1. Reload all state from persistence (kill switch, pause state).
2. Re-establish data feed connections.
3. Run at least one full cycle of the scoring engine.
4. Verify regime classification is current.

Sending signals before these steps complete risks sending stale, inconsistent, or corrupt data. The 5-minute window provides sufficient time for a full warm-up cycle.

### Deferred Message Handling

- Messages deferred during quiet mode are placed in a priority queue sorted by composite score (highest first).
- When quiet mode ends, deferred messages are re-evaluated against ALL gates (staleness, consistency, score floor, rate).
- Messages that fail re-evaluation are discarded (they were stale or the regime changed).
- Messages that pass are sent in priority order, respecting rate limits.

## 11. Rule 8: Kill Switch Persistence

### Statement

Kill switch and pause state must be persisted to disk/DB. On restart, restore previous kill/pause state.

### Specification

```
PERSISTED STATE FILE: /data/nzt48_output_state.json

SCHEMA:
{
    "kill_switch": {
        "active": boolean,
        "activated_at": ISO8601 timestamp or null,
        "activated_by": "operator" | "system" | "spam_kill",
        "reason": string or null
    },
    "pause_state": {
        "telegram_paused": boolean,
        "telegram_pause_until": ISO8601 timestamp or null,
        "pdf_paused": boolean,
        "warroom_paused": boolean
    },
    "last_persisted": ISO8601 timestamp,
    "engine_version": string
}
```

### Persistence Rules

| Event | Action |
|---|---|
| Kill switch activated | Write state to disk immediately (fsync) |
| Kill switch deactivated | Write state to disk immediately (fsync) |
| Pause state changed | Write state to disk immediately (fsync) |
| Spam kill triggered | Write pause state to disk immediately (fsync) |
| Every 60 seconds | Write current state to disk (periodic backup) |
| Service start | Read state from disk; restore all flags |

### Restore-on-Start Behaviour

```
FUNCTION restore_persisted_state():
    IF file_exists("/data/nzt48_output_state.json"):
        state = load_json("/data/nzt48_output_state.json")

        IF state.kill_switch.active:
            kill_switch.activate(reason="Restored from persistence: " + state.kill_switch.reason)
            log.warning("OUTPUT_POLICY: Kill switch restored as ACTIVE from persistence")
            notify_operator("Kill switch was active before restart — still active.")

        IF state.pause_state.telegram_paused:
            IF state.pause_state.telegram_pause_until > NOW():
                telegram.pause(until=state.pause_state.telegram_pause_until)
                log.info("OUTPUT_POLICY: Telegram pause restored — expires {until}")
            ELSE:
                log.info("OUTPUT_POLICY: Telegram pause expired during downtime — cleared")

        log.info("OUTPUT_POLICY: State restored from persistence",
            kill_active=state.kill_switch.active,
            last_persisted=state.last_persisted)

    ELSE:
        log.warning("OUTPUT_POLICY: No persisted state found — starting with defaults (all enabled)")
        write_default_state()
```

### Failure Handling

- If the state file is corrupt (invalid JSON, missing required fields), log a `CRITICAL` error, start with kill switch ACTIVE (fail-closed), and notify the operator.
- If the state file is missing, start with all outputs enabled (default state) and log a warning.
- If the disk write fails, retry 3 times with exponential backoff (1s, 2s, 4s). If all retries fail, log `CRITICAL` and continue operating with in-memory state only.

## 12. Failure Modes Summary

| Gate | Failure Mode | Default Action | Operator Notification |
|---|---|---|---|
| **Fail-Closed (R1)** | Quality check exception | SUPPRESS all Telegram; WATERMARK PDF/WR | Yes — Telegram alert |
| **Fail-Closed (R1)** | Quality engine unresponsive | SUPPRESS all Telegram; DEFER PDF | Yes — Telegram alert |
| **Staleness (R2)** | Data field exceeds TTL | BLOCK the specific message | No (logged only) |
| **Staleness (R2)** | Timestamp missing on field | BLOCK (treat as infinitely stale) | Yes — log warning |
| **Magnitude (R3)** | Return exceeds threshold | HOLD for manual review | Yes — Telegram alert |
| **Magnitude (R3)** | Return data unavailable | Treat as 0% (no flag) | No |
| **Consistency (R4)** | Regime mismatch | BLOCK signal; re-queue once | No (logged only) |
| **Consistency (R4)** | Regime engine null | BLOCK all signals | Yes — Telegram alert |
| **Score Floor (R5)** | Score < 10 | SUPPRESS | No (DEBUG log only) |
| **Score Floor (R5)** | Score unavailable | SUPPRESS (treat as 0) | Yes — log warning |
| **Rate Gov (R6)** | Per-minute exceeded | QUEUE | No |
| **Rate Gov (R6)** | Spam kill triggered | 15-min AUTO-PAUSE | Yes — Telegram alert |
| **Rate Gov (R6)** | 3x spam kill in 24h | KILL SWITCH activated | Yes — Telegram alert |
| **Restart (R7)** | During quiet mode | DEFER Telegram + PDF | War Room banner |
| **Restart (R7)** | State file corrupt | Start with kill switch ACTIVE | Yes — Telegram alert |
| **Kill Switch (R8)** | Disk write failure | Continue in-memory | Yes — log CRITICAL |
| **Kill Switch (R8)** | State file missing | Start with defaults (all enabled) | Yes — log warning |

### Universal Default

**If any gate encounters an unhandled exception, the default action is SUPPRESS.** The system never sends on uncertainty.

## 13. Operator Actions and Overrides

### Override API

All overrides require an authenticated API call with an API key and a mandatory reason field. Every override is logged to an immutable audit trail.

**Base endpoint:** `POST /api/override/{gate}`

**Required headers:**

```
X-API-Key: {operator_api_key}
Content-Type: application/json
```

**Required body fields:**

```json
{
    "reason": "string (mandatory, min 10 characters)",
    "duration_seconds": "integer (optional, for time-limited overrides)",
    "signal_id": "string (optional, for signal-specific overrides)"
}
```

### Per-Gate Override Endpoints

| Gate | Endpoint | Effect |
|---|---|---|
| Fail-Closed (R1) | `POST /api/override/quality-bypass` | Allow Telegram sends despite quality failure for `duration_seconds` |
| Staleness (R2) | `POST /api/override/staleness-bypass` | Allow sends with stale data for `duration_seconds` |
| Magnitude (R3) | `POST /api/override/magnitude-release` | Release a specific held signal by `signal_id` |
| Magnitude (R3) | `POST /api/override/magnitude-discard` | Permanently discard a specific held signal |
| Consistency (R4) | `POST /api/override/regime-bypass` | Allow regime-mismatched signals for `duration_seconds` |
| Score Floor (R5) | `POST /api/override/score-bypass` | Temporarily lower score floor to specified value |
| Rate Gov (R6) | `POST /api/override/rate-reset` | Reset rate counters and clear spam kill pause |
| Kill Switch (R8) | `POST /api/override/kill-switch-off` | Deactivate kill switch |
| Kill Switch (R8) | `POST /api/override/kill-switch-on` | Activate kill switch |

### Override Audit Log Schema

```json
{
    "timestamp": "2026-02-27T09:35:00Z",
    "gate": "magnitude",
    "action": "release",
    "signal_id": "sig_20260227_093115_QQQ3L",
    "operator_key_hash": "sha256:abc123...",
    "reason": "Confirmed genuine move via LSE feed — corporate earnings beat",
    "duration_seconds": null,
    "ip_address": "10.0.0.1"
}
```

### Override Constraints

- Overrides are **time-limited by default**. If `duration_seconds` is not provided, the override expires after 300 seconds (5 minutes).
- Maximum override duration: 3600 seconds (1 hour). Longer overrides require a second API call to extend.
- Overrides do not stack. A new override replaces the previous one for the same gate.
- Override audit logs are append-only and cannot be deleted or modified.
- The API key must be a 256-bit key stored in environment variable `NZT_OVERRIDE_API_KEY`. It is never logged in plaintext.

## 14. Acceptance Tests

### Rule 1: Fail-Closed Default

| Test ID | Scenario | Expected Result | Pass Criteria |
|---|---|---|---|
| R1-T01 | Disable price feed; trigger signal | Telegram signal suppressed | No Telegram message sent; suppression logged |
| R1-T02 | Disable price feed; trigger PDF generation | PDF generated with DEGRADED DATA watermark | Watermark visible on all pages |
| R1-T03 | Disable price feed; check War Room | War Room displays DEGRADED DATA banner | Banner visible and persistent |
| R1-T04 | Restore price feed; trigger signal | Telegram signal sent normally | Signal delivered; no watermark/banner |
| R1-T05 | Quality engine process crashes | All Telegram suppressed; PDF deferred | No outbound messages until quality engine recovers |

### Rule 2: Staleness Gate

```json
{
    "timestamp": "ISO8601",
    "level": "DEBUG|INFO|WARNING|ERROR|CRITICAL",
    "component": "output_policy",
    "gate": "fail_closed|staleness|magnitude|consistency|score_floor|rate_gov|restart|kill_switch",
    "action": "PASS|BLOCK|SUPPRESS|HOLD|DEFER|QUEUE",
    "signal_id": "string or null",
    "ticker": "string or null",
    "details": {}
}
```
